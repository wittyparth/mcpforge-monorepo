## Conventions for New Code (must follow)

### Python (apps/api)

- **Async everywhere.** No sync DB calls, no sync HTTP. Use `httpx.AsyncClient` for outbound.
- **Type hints everywhere.** Run `mypy --strict`. New functions must have full type annotations, including return type.
- **No `Any`.** If you don't know the type, use `Unknown` + a type guard.
- **Services take `AsyncSession` in constructor**, not as method args. Same for repositories.
- **Routes do HTTP plumbing only.** Business logic in services. DB queries in repositories.
- **Errors are typed.** Raise `AppError` subclasses (`NotFoundError`, `UnauthorizedError`, `ValidationError`, `RateLimitError`, `UpstreamError`, etc.). They map to HTTP via `core/exceptions.py` exception handlers.
- **Logging via `structlog`.** `logger = get_logger(__name__)`. Log structured key-value pairs: `logger.info("server_created", server_id=str(s.id), slug=s.slug)`.
- **No `print()`.** No `os.environ` reads. Use `settings.X` from `core.config`.
- **No `as any` / `@ts-ignore` in TS** either.
- **Tests use the same `conftest.py` fixtures.** Add a new fixture if you need shared state.

### TypeScript (apps/web)

- **Server Components by default.** Add `"use client"` only when you need hooks, state, browser APIs.
- **Data fetching via TanStack Query** for client-side. Server Components for SSR data.
- **Forms: react-hook-form + Zod** (always). Define Zod schema in `lib/validators.ts`, derive the form type from it.
- **Components in `components/ui/`** for primitives (shadcn), `components/<domain>/` for domain components.
- **Hooks in `hooks/use-<resource>.ts`**, one per resource, using TanStack Query.
- **Types in `types/api.ts`** for hand-written, `@mcpforge/shared-types` for generated.
- **Path alias: `@/*` → `src/*`**.
- **No localStorage for tokens.** Backend sets httpOnly cookies, frontend uses `credentials: 'include'`.
- **No `any`**, no `@ts-ignore`. Use `unknown` + narrowing.
- **Tailwind v4** for styling. No CSS modules. Use CSS variables in `globals.css` for theming.

### Database migrations

- **Alembic, autogenerate when possible.** Always review the generated migration.
- **Naming: `NNNN_description.py`** (e.g., `0002_add_teams.py`).
- **Reversible.** Provide both `upgrade()` and `downgrade()`.
- **No data loss in migrations.** Adding a column = nullable or with default. Removing a column = rename first, drop later.
- **Test migrations** by running `alembic upgrade head` then `alembic downgrade -1` on a fresh DB.

### Git / commits

- No commits without explicit user request.
- Conventional Commits: `feat:`, `fix:`, `chore:`, `docs:`, `refactor:`, `test:`, `build:`, `ci:`.
- One commit per logical change. Don't bundle 5 features.
