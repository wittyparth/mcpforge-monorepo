# arxiv 2602.18914 — The MCP Description Quality Study

> **For AI agents:** This is the academic foundation of our AI Description Engine (F2). The 4 quality dimensions, the 260% selection lift number, the smell categories — all come from this paper. Use this when designing prompts, scoring, or explaining the feature to users.

---

## 1. Full Citation

**Title:** "From Docs to Descriptions: Smell-Aware Evaluation of MCP Server Descriptions"
**Authors:** Peiran Wang, Ying Li, Yuqiang Sun, Chengwei Liu, Yang Liu, Yuan Tian
**Institutions:** UCLA & NTU
**Paper:** https://arxiv.org/abs/2602.18914
**Date:** February 21, 2026
**Subject:** cs.SE (Software Engineering)

## 2. TL;DR — The 4 Quality Dimensions

| Dimension | Impact on Selection | What it means |
|---|---|---|
| **Functionality** | **+11.6%** (p<0.001) | Does the description say what the tool does and when to use it? |
| **Accuracy** | **+8.8%** (p<0.001) | Are parameter types, constraints, and behaviors described correctly? |
| **Information Completeness** | **+5.9%** (p<0.01) | Are all parameters (including optional ones) and the return value described? |
| **Conciseness** | **+1.5%** (p<0.05) | Does the description avoid irrelevant detail clutter? |

**Functionality is the strongest single predictor of selection.** Accuracy and Completeness are close seconds.

## 3. The 260% Number — Exact Source

**Finding IV (Section 5.4):** "Across all 10 server groups, the standard-compliant server achieved an average selection probability of 72%, compared to the 20% baseline expected under uniform random selection. This represents a **260% relative increase** in selection probability."

**Methodology detail:** Five functionally equivalent MCP servers competed for tool selection by an LLM judge. The one with standard-compliant descriptions won 72% of the time (vs 20% uniform random). Consistent across all 10 domains (65%–81%) and all query complexity levels (68%–78%).

**What this means for MCPForge:** A well-described tool is selected 3.6x more often than a poorly-described one. Our AI Engine's job is to make every tool well-described.

## 4. 18 Smell Categories Identified

The paper found 18 specific description smell patterns across 10,831 servers analyzed:

### 4.1 Top findings (with counts)
- **73%** of servers have repeated tool names
- **3,449** servers have wrong parameter meanings
- **3,093** servers lack return value descriptions
- **1,285** servers have missing parameter descriptions
- **2,904** servers have irrelevant detail clutter

### 4.2 The "Code-First, Description-Last" Pattern
The paper calls out a widespread pattern where developers focus on implementation and treat descriptions as an afterthought. This is MCPForge's core opportunity — **automated description remediation**.

## 5. Methodology

### 5.1 Sample
- **10,831** MCP servers collected from GitHub, MCP.so, Glama, PulseMCP, Smithery
- **Languages:** Python (3,221), TypeScript (2,908), JS/TS mixed (1,556), other
- **Extraction:** AST-based pattern matching for all tool registration methods

### 5.2 Quality dimension derivation
- LLM-based incremental card-sorting used to derive smell categories
- 7 major quality standard types analyzed (ISO/IEC 25010, OWASP, etc.)
- 4 dimensions emerged as the most predictive

### 5.3 Mutation study
- Systematically mutated descriptions to measure LLM tool selection impact
- Each dimension tested independently
- Statistical significance verified (p-values reported)

### 5.4 Competitive selection study
- 5 functionally equivalent servers competed
- LLM judge selected among them
- 10 server groups, multiple query complexities
- Average 72% selection for standard-compliant vs 20% baseline

## 6. The 4 Dimensions in Detail (with examples)

### 6.1 Functionality (+11.6% impact)
**What:** Does the description convey what the tool does AND when to use it?
**Strong signals:**
- "USE THIS WHEN you need X"
- Action verb at the start ("Retrieve", "Create", "Update")
- "DO NOT USE FOR Y" (disambiguation)
- Returns clause
**Weak signals:**
- Vague verbs ("Handle", "Process", "Do")
- No trigger condition
- Single-word descriptions

**Example transformation:**
- ❌ "get" → ✅ "Retrieve user profile by ID. USE THIS WHEN you have a specific user ID and want their profile. Returns user details or 404."

### 6.2 Accuracy (+8.8% impact)
**What:** Do parameter names, types, and constraints match actual behavior?
**Strong signals:**
- Exact type info ("string, ISO 8601")
- Valid value ranges ("1-100", "must be positive")
- Format hints ("UUID v4", "URL-encoded")
**Weak signals:**
- "An ID" (which format?)
- "Some data" (what type?)
- Parameter names that don't match the API

**Example transformation:**
- ❌ `{ "id": "string" }` → ✅ `{ "id": "string (UUID v4, example: 'usr_abc123')" }`

### 6.3 Information Completeness (+5.9% impact)
**What:** Are ALL parameters (including optional) AND the return value described?
**Strong signals:**
- Every parameter has a description
- Optional parameters explicitly described
- Return value structure + edge cases (empty, error)
**Weak signals:**
- "Various parameters" (which?)
- No return value description
- Optional params skipped

**Example transformation:**
- ❌ (no return description) → ✅ "Returns JSON array of products; empty array if no matches; 404 error if category doesn't exist."

### 6.4 Conciseness (+1.5% impact)
**What:** Does the description avoid irrelevant detail clutter?
**Strong signals:**
- 50-300 character range
- Focused on essentials
- Action-oriented
**Weak signals:**
- Bloated descriptions (>500 chars)
- Marketing speak ("Our world-class API...")
- Implementation details that don't help the LLM

**Example transformation:**
- ❌ "This endpoint leverages our state-of-the-art retrieval system to fetch data from the user's account using the most appropriate strategy based on context..." → ✅ "Retrieve user data by ID."

## 8. Production Server Description Patterns (per the paper + CallSphere)

### 8.1 Best practices
1. **"USE THIS WHEN"** explicit trigger
2. **"DO NOT USE FOR"** disambiguation
3. **`readOnlyHint` for non-destructive tools**
4. **2-4 sentences per tool**
5. **`verb_noun` naming** (`create_customer`, not `createCustomer` or `CustomerCreate`)
6. **Return value description with edge cases**
7. **No version suffixes** in tool names

### 8.2 Anti-patterns
- Single-word names ("get", "create")
- Marketing prose
- Implementation details
- Versioned names (`FetchResources_v2`)
- Repeated names (LLMs can't distinguish)
- Missing return descriptions
- Vague parameter docs

## 9. How MCPForge Applies This

### 9.1 AI Description Engine (F2)
- System prompt explicitly defines the 4 dimensions with scoring rubrics
- User prompt includes the tool, sibling tools (for disambiguation), and few-shot examples
- Output is structured JSON with name, description, parameters, return_description
- Quality scorer (heuristic, not LLM) scores on the 4 dimensions

### 9.2 Quality Score Badges
- **90-100:** Green "Excellent" badge
- **70-89:** Yellow "Good" badge
- **50-69:** Orange "Fair" badge — warning shown
- **<50:** Red "Poor" badge — blocking warning

### 9.3 Improvements Display
Each tool's review panel shows orange badges listing the improvements made:
- "Added disambiguation"
- "Added return value description"
- "Rewrote 3 parameter descriptions"
- "Renamed to 'search_products'"

### 9.4 Anti-Pattern Detection
The scanner (F5) detects:
- Tool names matching `^(get|create|update|delete)$` (too vague)
- Descriptions shorter than 30 chars
- Missing return value descriptions
- Identical tool names (collision detection)

## 10. Limitations of the Paper

1. **LLM judge used, not human eval** — selection probability is what the LLM thinks is best
2. **5-server competitive setting** — may not reflect 100-server real-world use
3. **Mostly English descriptions** — multilingual support untested
4. **One-shot selection** — multi-turn agent behavior may differ
5. **No replication studies yet** (as of Feb 2026) — we should run internal validation
